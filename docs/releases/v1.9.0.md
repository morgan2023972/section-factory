## Changements principaux

### 1) Stabilisation runtime du CLI repair

Le CLI `repair` valide desormais la section d entree puis:

- evite tout appel au moteur de reparation si aucun issue bloquant n est detecte
- conserve le comportement d ecriture uniquement en cas de succes complet avec `--write`

Cela supprime les tentatives inutiles et aligne le comportement avec les attentes de production.

### 2) Observabilite legere (sans bruit)

Le rapport `text` et le JSON incluent un resume minimal:

- validation initiale: `OK` / `FAIL`
- repair tente: `yes` / `no`
- resultat final utilise: `yes` / `no`
- amelioration detectee: `yes` / `no`

Le format JSON expose ces informations dans un bloc `observability`.

### 3) Comportement repair documente dans le core

Ajout d une note interne dans `repairSection.ts` pour expliciter:

- ce qui est bien corrige (schema, desequilibres Liquid simples, nettoyage local)
- ce qui peut rester partiel (schema JSON complexe)
- les limites (pas de reecriture semantique lourde CSS/JS)

### 4) Renforcement et nettoyage des tests

- Organisation explicite par groupes (Safety, amelioration, validation, non-destruction, idempotence).
- Idempotence verifiee apres normalisation sur 2 fixtures representatives:
  - `missing-schema-block.liquid`
  - `unclosed-liquid-if.liquid`
- Helpers communs extraits dans `tests/integration/repairTestUtils.ts`.
- Nommage harmonise entre baseline comportementale et review fixtures.
